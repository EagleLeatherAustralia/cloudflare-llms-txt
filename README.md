# cloudflare-llms-txt

Surfaces `llms.txt`, `llms-full.txt`, `security.txt`, and a read-only **MCP server** at the apex of **eagleleather.com.au** via a Cloudflare Worker. `llms.txt` and `security.txt` are bundled directly into the Worker; `llms-full.txt` is proxied (with edge caching) from the Magento media directory; `/mcp` parses that same feed and exposes search/lookup tools to MCP clients.

The point of this project: give AI agents a sanctioned, low-cost crawl path so they stop hammering the Vue Storefront `_search` and `render-list` endpoints to extract the catalog. The security.txt is along for the ride — same Worker, same deploy.

---

## Background

Cloudflare AI bot analytics on `www.eagleleather.com.au` showed AI crawlers paginating the VSF1 Elasticsearch endpoints to dump the catalog (~25 MB on `_search`, ~17 MB on `render-list`, plus 2.6 GB/month of full storefront HTML page renders). These endpoints exist for the frontend JS, not bulk extraction.

We don't want to flat-out block — we want to encourage agentic shopping (ChatGPT shopping, Claude with computer-use, Perplexity Shopping, etc.). Those agents act on behalf of a user and make narrow transactional calls. **Shape, don't block** — distinguish bulk extraction from narrow transactional calls.

`llms.txt` and `llms-full.txt` are the sanctioned alternative: a hand-curated index file pointing agents at the product feed/sitemap, plus an auto-generated full content dump.

## Architecture

```
                          eagleleather.com.au
                                 │
                                 ▼
                         Cloudflare Worker
   (routes: /llms.txt, /llms-full.txt, /security.txt, /.well-known/security.txt,
            /mcp, /.well-known/mcp/server-card.json)
              │                                   │
              ▼                                   ▼
   llms.txt + security.txt              ${FEED_BASE} (Magento media/feeds dir)
   (bundled into the Worker)                └── llms-full.txt  (auto-generated by Xtento)
                                                    │
                                                    ▼
                                          /mcp parses the same feed
                                          (per-isolate memo + edge cache)
```

`llms.txt` and `security.txt` are bundled directly into the Worker as text modules — single source of truth in this repo, no upload step, no risk of word-wrap from `scp`. `llms-full.txt` is fetched from the Magento media directory because it's auto-generated nightly and too large to bundle.

Anything that isn't one of those bound paths flows untouched to the Magento origin — the Worker only binds the handful of paths listed in `worker/wrangler.toml`.

## Setup

```bash
cd ~/dev/cloudflare-llms-txt
npm install
npx wrangler login
```

Set `CF_ACCOUNT_ID` in your shell or in `worker/wrangler.toml`. Find it in the Cloudflare dashboard sidebar.

## Deploy

```bash
npm run deploy        # deploys the Worker (binds all served paths at apex + www)
```

The Worker reads `FEED_BASE` from a Worker secret (set once with `wrangler secret put FEED_BASE`). For local dev, copy `worker/.dev.vars.example` → `worker/.dev.vars` and fill in the value; `.dev.vars` is gitignored so it never reaches the repo.

## Local dev

## Updating `llms.txt`

Edit `llms.txt` at the repo root, then redeploy the Worker — the file is bundled into the Worker at build time, so no separate upload step:

```bash
npm run deploy
```

The Worker emits `Cache-Control: public, max-age=3600`, so existing cached responses can take up to an hour to refresh. Use the refresh endpoint below for instant invalidation.

## Force-refreshing the edge cache

The Worker has a built-in purge endpoint. Pass the refresh token as the `X-Refresh-Token` header on a request to any served path; the Worker calls the Cloudflare cache-purge API for every served URL (apex + www, all files):

```bash
curl -H "X-Refresh-Token: $REFRESH_TOKEN" https://eagleleather.com.au/llms.txt
```

Returns `200 OK` with body `OK` on success, `401` if the token doesn't match, `500` if the secrets aren't set, `502` if the purge API call fails.

> **Why a header instead of a query string?** Cloudflare Redirect Rules don't preserve query strings by default, and this zone has an apex→www redirect. A header survives the redirect; a query parameter would get dropped before the Worker ever sees it. The Worker still accepts `?refresh=<token>` as a fallback if you call the www host directly.

### Setup

All four secrets (`FEED_BASE`, `PURGE_API_TOKEN`, `PURGE_ZONE_ID`, `REFRESH_TOKEN`) are managed by `npm run deploy`. One-time setup:

```bash
cp worker/.dev.vars.example worker/.dev.vars
# edit worker/.dev.vars and fill in FEED_BASE, PURGE_API_TOKEN, PURGE_ZONE_ID
npm run deploy
```

`scripts/deploy.sh` reads `worker/.dev.vars` and pushes each non-empty value to Cloudflare via `wrangler secret put` before deploying. Then it rotates `REFRESH_TOKEN` to a fresh 64-char hex string and prints it. `.dev.vars` is gitignored — it never reaches the repo.

To rotate or change a stable secret later: edit `worker/.dev.vars` and re-run `npm run deploy`. Empty values are skipped (so a partial `.dev.vars` won't blank production secrets).

`PURGE_API_TOKEN` needs Zone:Cache Purge permission for the zone — create at https://dash.cloudflare.com/profile/api-tokens (template: "Custom token" → Permissions: Zone → Cache Purge → Purge → Resources: include the zone). `PURGE_ZONE_ID` is the zone ID from the dashboard sidebar.

Without these set, the refresh endpoint returns a 500 explaining what's missing; the normal served paths keep working regardless.

If you want to redeploy without rotating `REFRESH_TOKEN` (e.g. so an existing automated tool keeps working), use `npm run deploy:keep-token` — note this also skips the secret-sync step, so use it only between full deploys.

## Updating `llms-full.txt`

It's auto-generated by Xtento. To change the output format, edit `xtento-llms-full.xsl` and re-paste it into the Xtento profile in Magento admin (Catalog → Xtento Product Export → your `llms-full.txt` profile → Template). The profile runs on schedule and writes to `pub/media/feeds/llms-full.txt`.

To regenerate on demand: hit "Run profile now" in the Xtento UI.

## Updating `security.txt`

`security.txt` is bundled into the Worker the same way `llms.txt` is. Edit `security.txt` at the repo root, then `npm run deploy`. It's served at both `/.well-known/security.txt` (the RFC 9116 canonical path) and `/security.txt` (the legacy path some scanners still check).

**Annual renewal required.** RFC 9116 mandates an `Expires:` field, currently set to `2027-04-28T00:00:00Z`. Bump it on or before that date — researcher tooling flags expired files as untrustworthy. Set a calendar reminder.

If you ever publish a PGP key for encrypted reports, add an `Encryption:` line pointing at the public key URL. Not currently configured.

## Verifying

```bash
# Origin (only llms-full.txt is actually proxied — others are bundled)
curl -I "$FEED_BASE/llms-full.txt"

# Via the Worker (production)
curl -I https://eagleleather.com.au/llms.txt
curl -I https://eagleleather.com.au/llms-full.txt
curl -I https://eagleleather.com.au/.well-known/security.txt
curl -I https://eagleleather.com.au/security.txt
```

All four should return `200 text/plain; charset=utf-8` with `X-Served-By: cloudflare-llm-txt-worker`. The `/llms.txt` and `/llms-full.txt` responses also carry a `Link:` header advertising the MCP server (RFC 8288 / 9727 discoverability).

#### Markdown content negotiation

`/llms.txt` and `/llms-full.txt` are markdown-shaped already, but the Worker also honours `Accept: text/markdown` and switches the response `Content-Type` accordingly (with `Vary: Accept` so the edge cache keeps both variants separate). Lets agents that explicitly negotiate for markdown get the type they asked for instead of the default `text/plain`.

```bash
curl -sS -I https://www.eagleleather.com.au/llms.txt                                # → text/plain
curl -sS -I -H 'Accept: text/markdown' https://www.eagleleather.com.au/llms.txt     # → text/markdown
curl -sS -I -H 'Accept: text/markdown' https://www.eagleleather.com.au/llms-full.txt # → text/markdown
```

### MCP server

```bash
# Inspect with the official MCP Inspector (live deploy)
npx @modelcontextprotocol/inspector https://www.eagleleather.com.au/mcp
# → choose "Streamable HTTP", connect, browse the 5 tools

# JSON-RPC smoke test
curl -sS https://www.eagleleather.com.au/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'

curl -sS https://www.eagleleather.com.au/mcp \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json, text/event-stream' \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"search_products","arguments":{"query":"latex hood","limit":3}}}'
```

Tools exposed (all read-only, no auth):

| Tool | Purpose |
|---|---|
| `search_products` | free-text search across name/SKU/brand/description, optional category/brand/in_stock filters |
| `get_product` | full record by SKU |
| `check_stock` | lightweight availability + price by SKU |
| `list_categories` | every breadcrumb in the catalogue |
| `list_brands` | brand list with product counts |

Backed by `llms-full.txt` (parsed once per isolate, refreshed every hour). Write tools (`add_to_cart`, `checkout`) are tracked in *Future work* — they require Magento auth/order plumbing.

#### MCP Server Card

For agent discovery (per the draft [MCP Server Card spec, SEP-1649](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2127)), the Worker also serves a small JSON card at `/.well-known/mcp/server-card.json` with `serverInfo`, `transport`, and `capabilities`. The schema is still in flux upstream so we keep the card minimal — when SEP-1649 merges, update the response in `worker/src/index.ts` to match the final field names.

```bash
curl https://www.eagleleather.com.au/.well-known/mcp/server-card.json
```

### Optional: homepage Link header

Tell agents about the catalogue/MCP from the homepage without routing homepage traffic through this Worker. In the Cloudflare dashboard, **Rules → Transform Rules → Modify Response Header → Add**:

- **When**: `(http.host eq "www.eagleleather.com.au" and http.request.uri.path eq "/")`
- **Set static**: `Link` = `</llms.txt>; rel="llms-txt", </llms-full.txt>; rel="describedby"; type="text/plain", </mcp>; rel="mcp-server"`

## Cloudflare dashboard setup checklist

Things to configure in the dashboard (one-off, no code). Apply to the `eagleleather.com.au` zone unless noted.

> **Dashboard versions.** Cloudflare is mid-migration between an older IA (under "Security → WAF") and a newer one (under "Security → Security rules"). Both routes still work; the steps below describe the new IA with the old path noted underneath. The deep-link URLs (replace `:account` with your account ID and `:zone` with `eagleleather.com.au`) redirect to whichever version your account is on.

### 1. Rate Limiting Rules — required after MCP deploy

- **Deep-link**: `https://dash.cloudflare.com/?to=/:account/:zone/security/security-rules` → **Create rule → Rate limiting rule**
- **Old IA**: Security → WAF → Rate limiting rules → Create rule
- **Docs**: https://developers.cloudflare.com/waf/rate-limiting-rules/create-zone-dashboard/

Create one rule per row:

| Rule name | Expression | Counting characteristic | Rate | Action | Mitigation timeout |
|---|---|---|---|---|---|
| `mcp-rate-limit` | `(http.host in {"eagleleather.com.au" "www.eagleleather.com.au"} and http.request.uri.path eq "/mcp")` | IP source address | 60 / 1 minute | Block | 60 seconds |
| `llms-full-rate-limit` | `(http.host in {"eagleleather.com.au" "www.eagleleather.com.au"} and http.request.uri.path eq "/llms-full.txt")` | IP source address | 10 / 1 minute | Block | 60 seconds |

A heavy agent session is ≈ 5–20 tool calls, so 60/min/IP is generous for legitimate use and tight for runaway loops or scrapers. `/llms.txt` and `/security.txt` are tiny and infinitely cacheable — leave them unlimited.

### 2. Response Header Transform Rule — homepage `Link` header (optional)

- **Deep-link**: `https://dash.cloudflare.com/?to=/:account/:zone/rules/overview` → **Create rule → Response Header Transform Rule**
- **Old IA**: Rules → Transform Rules → Modify Response Header → Create rule
- **Docs**: https://developers.cloudflare.com/rules/transform/response-header-modification/create-dashboard/

Settings:

- **Rule name**: `homepage-link-discovery`
- **When incoming requests match (Custom filter expression)**: `(http.host in {"eagleleather.com.au" "www.eagleleather.com.au"} and http.request.uri.path eq "/")` — matches both apex (the 301 redirect response) and www (the HTML response)
- **Modify response header**: *Set static*
- **Header name**: `Link`
- **Value**: `</llms.txt>; rel="llms-txt", </llms-full.txt>; rel="describedby"; type="text/plain", </mcp>; rel="mcp-server"`

This advertises the agent endpoints from the homepage without routing homepage traffic through the Worker.

### 3. Cache settings — verify only

The Worker emits `Cache-Control` itself, so no Cache Rule is needed. But under **Caching → Configuration**, confirm **Browser Cache TTL** is set to *Respect Existing Headers* — otherwise the CDN rewrites our TTLs.

### 4. Bot Fight Mode — keep OFF zone-wide

- **Deep-link**: `https://dash.cloudflare.com/?to=/:account/:zone/security/settings` → filter **Bot traffic** → toggle **Bot Fight Mode** off
- **Old IA**: Security → Bots → Bot Fight Mode → Off
- **Docs**: https://developers.cloudflare.com/bots/get-started/bot-fight-mode/

**Important caveat verified against current docs:** Bot Fight Mode (free plan) **cannot be skipped per-path**. Neither WAF Custom Rules with the *Skip* action nor Page Rules can bypass it — only **Super Bot Fight Mode** (Pro/Business/Enterprise) supports per-path skip.

So on the free plan you have two options:

1. **Recommended**: keep Bot Fight Mode off zone-wide. The site already declares its bot policy via `robots.txt` and `Content-Signal: search=yes,ai-train=no`.
2. Upgrade to a paid plan with Super Bot Fight Mode, then add a WAF Custom Rule (Security → Security rules → Create rule → Custom rule) with action *Skip → All remaining custom rules + Super Bot Fight Mode* for `(http.request.uri.path in {"/mcp" "/llms.txt" "/llms-full.txt"})`.

### 5. Worker secrets — managed by `npm run deploy`

`scripts/deploy.sh` syncs `FEED_BASE`, `PURGE_API_TOKEN`, `PURGE_ZONE_ID`, and rotates `REFRESH_TOKEN` automatically. To inspect or rotate manually:

- **Deep-link**: `https://dash.cloudflare.com/?to=/:account/workers-and-pages` → **eagleleather-llms-router** → **Settings → Variables and Secrets**

### 6. Workers route conflicts — verify before first deploy

- **Deep-link** (per-Worker view): `https://dash.cloudflare.com/?to=/:account/workers-and-pages` → **eagleleather-llms-router → Settings → Domains & Routes**
- **Docs**: https://developers.cloudflare.com/workers/configuration/routing/routes/

Make sure no other Worker, Page Rule, or Redirect Rule already binds any of:
- `/llms.txt`
- `/llms-full.txt`
- `/security.txt`
- `/.well-known/security.txt`
- `/mcp`

Existing rules win and silently shadow the deploy — remove conflicts first.

## Troubleshooting

**Worker route conflict** — if `eagleleather.com.au/llms.txt` is already routed elsewhere (e.g. an existing Page Rule), Cloudflare will reject the route. Remove the conflicting rule from the dashboard first.

**5xx from the Worker** — the origin behind `$FEED_BASE` isn't reachable. Check that the Magento `pub/media/feeds/` directory is publicly served and not behind admin IP allowlisting. `npm run tail` will show the upstream errors live.

**Stale content** — the Worker caches at edge for 1h. Either wait, change `cacheTtl` in `worker/src/index.ts`, or purge from the Cloudflare dashboard.
